# NOTICES_TEI_TARGET.md

## 1. Objet

Ce document complète :

- `NOTICES_CHAIN_SPEC.md`
- `WORD_STYLES_MAPPING.md`
- `NOTICES_VALIDATION_RULES.md`

Il fixe la **forme TEI cible** produite par la chaîne d’import des **notices Word** dans Ekdosis–TEI Studio (ETS).

Son but est de fournir à l’implémentation une cible XML **stable, lisible, minimaliste et testable**, afin d’éviter toute improvisation structurelle lors de la conversion.

Il s’applique uniquement à la chaîne des **notices**, pas à celle des pièces de théâtre.

---

## 2. Principe général

La conversion d’une notice ne doit pas produire un XML « libre », mais un document TEI conforme à un **profil de sortie maison**.

Ce profil suit les principes suivants :

- structure simple et régulière ;
- nombre limité d’éléments TEI ;
- hiérarchie explicite ;
- conservation des informations éditorialement utiles ;
- absence d’encodages savants ou expérimentaux hors périmètre V1.

La cible TEI doit donc être :

- suffisamment expressive pour les notices académiques ;
- suffisamment simple pour être robuste ;
- suffisamment stable pour être testée automatiquement.

---

## 3. Niveau de sortie attendu

### 3.1. Type de sortie

La sortie attendue est un **document TEI complet**, et non un simple fragment XML.

La structure minimale attendue est :

```xml
<TEI xmlns="http://www.tei-c.org/ns/1.0">
  <teiHeader>...</teiHeader>
  <text>
    <body>
      <div type="notice" xml:id="...">
        ...
      </div>
    </body>
  </text>
</TEI>
```

### 3.2. Politique générale

Le module peut produire un `teiHeader` minimal en V1, mais il doit toujours produire :

- un élément racine `<TEI>` ;
- un `<teiHeader>` ;
- un `<text>` ;
- un `<body>` ;
- un `<div type="notice">` contenant la notice.

---

## 4. Structure TEI cible globale

## 4.1. Squelette minimal

```xml
<TEI xmlns="http://www.tei-c.org/ns/1.0">
  <teiHeader>
    <fileDesc>
      <titleStmt>
        <title>...</title>
      </titleStmt>
      <publicationStmt>
        <p>Generated by Ekdosis–TEI Studio</p>
      </publicationStmt>
      <sourceDesc>
        <p>Imported from a styled Word notice.</p>
      </sourceDesc>
    </fileDesc>
  </teiHeader>
  <text>
    <body>
      <div type="notice" xml:id="notice-slug">
        <head type="main">...</head>
        ...
      </div>
    </body>
  </text>
</TEI>
```

## 4.2. Contraintes minimales

Le document généré doit toujours contenir :

- exactement un `<div type="notice">` racine dans le corps ;
- exactement un `<head type="main">` dans ce `div` ;
- zéro ou un `<head type="sub">` placé immédiatement après le titre principal ;
- une hiérarchie de `<div>` imbriqués pour les niveaux de section.

---

## 5. Politique sur `xml:id`

### 5.1. `xml:id` du div de notice

Le `div` racine de la notice doit porter un `xml:id` stable.

Forme recommandée :

```xml
<div type="notice" xml:id="berenice-notice">
```

### 5.2. Source de l’identifiant

L’identifiant peut être construit à partir :

- du titre principal normalisé ;
- ou d’un slug fourni par ETS ;
- ou d’une stratégie interne centralisée.

### 5.3. Règle de stabilité

À contenu égal, la génération d’identifiant doit être stable.

---

## 6. Mapping des styles de structure

## 6.1. Titre principal

Le style Word `Titre` / `Title` doit être converti en :

```xml
<head type="main">Bérénice</head>
```

### Règles

- obligatoire ;
- unique à l’échelle de la notice cible ;
- contenu textuel inline autorisé.

## 6.2. Sous-titre

Le style `Sous-titre` / `Subtitle` doit être converti en :

```xml
<head type="sub">Notice</head>
```

### Règles

- facultatif ;
- placé immédiatement après `<head type="main">` ;
- non répété.

## 6.3. Titres de niveau 1 à 4

Les styles Word `Titre 1` à `Titre 4` doivent être convertis en hiérarchie de `div` imbriqués.

### Mapping canonique

- `Titre 1` -> `<div type="section">`
- `Titre 2` -> `<div type="subsection">`
- `Titre 3` -> `<div type="subsubsection">`
- `Titre 4` -> `<div type="level4">`

Chaque `div` de section commence par un `<head>` simple.

### Exemple

```xml
<div type="section">
  <head>Genèse</head>
  <p>...</p>
  <div type="subsection">
    <head>Datation</head>
    <p>...</p>
  </div>
</div>
```

### Règles

- pas de texte libre directement entre deux niveaux logiques sans conteneur adéquat ;
- toute section ouverte doit être refermée proprement ;
- aucune création de niveaux supplémentaires hors `section`, `subsection`, `subsubsection`, `level4`.

---

## 7. Paragraphes

## 7.1. Paragraphe courant

Le style `Normal` produit :

```xml
<p>...</p>
```

## 7.2. Paragraphe sans retrait

Le style natif Word retenu comme **style sans retrait** produit :

```xml
<p rend="noindent">...</p>
```

### Règles

- le style sans retrait reste un paragraphe ordinaire du point de vue structurel ;
- il se distingue uniquement par `@rend="noindent"` ;
- aucune autre valeur de `@rend` ne doit être ajoutée en V1, sauf besoin explicitement documenté.

---

## 8. Citations

Le style Word `Citation` / `Quote` produit un bloc de citation.

### Forme cible

```xml
<quote>
  <p>Texte cité...</p>
</quote>
```

### Règles

- un paragraphe `Citation` = un bloc `<quote>` contenant un `<p>` ;
- plusieurs paragraphes `Citation` consécutifs peuvent produire soit plusieurs `<quote>`, soit un seul `<quote>` contenant plusieurs `<p>`, mais l’implémentation doit choisir une stratégie unique et la garder stable ;
- recommandation V1 : **un bloc `<quote>` par paragraphe source**, pour simplifier la prédictibilité.

---

## 9. Mises en forme inline

Les effets de caractère autorisés doivent être convertis en `<hi>`.

### Mapping canonique

- italique -> `<hi rend="italic">...</hi>`
- gras -> `<hi rend="bold">...</hi>`
- petites capitales -> `<hi rend="smallcaps">...</hi>`
- exposant -> `<hi rend="sup">...</hi>`
- indice -> `<hi rend="sub">...</hi>`

### Exemple

```xml
<p>Texte avec <hi rend="italic">italique</hi>, <hi rend="bold">gras</hi> et <hi rend="smallcaps">petites capitales</hi>.</p>
```

### Règles

- les petites capitales ne doivent jamais devenir un signal structurel ;
- les effets inline peuvent être imbriqués si la source le justifie ;
- l’implémentation doit éviter les `<hi>` vides ;
- les séquences contiguës de même `@rend` doivent être fusionnées si cela ne change pas le contenu.

---

## 10. Notes

Les notes de bas de page Word natives doivent produire :

```xml
<note place="foot">...</note>
```

### Exemple

```xml
<p>Texte de paragraphe<note place="foot">Contenu de la note.</note>.</p>
```

### Règles

- la note est insérée inline à l’endroit de son appel ;
- le contenu de note peut contenir des paragraphes ou du contenu inline selon la conversion disponible ;
- en V1, une note simple textuelle suffit ;
- aucune note de fin n’est attendue.

---

## 11. Listes

Les listes Word natives doivent produire une structure TEI simple.

### Forme cible

```xml
<list>
  <item>Premier point</item>
  <item>Deuxième point</item>
</list>
```

### Listes numérotées

Deux stratégies sont acceptables, mais il faut en choisir une seule pour la V1 :

1. ne pas typer la liste ;
2. ajouter `type="ordered"` pour les listes numérotées.

### Recommandation V1

```xml
<list type="ordered">
  <item>...</item>
</list>
```

pour les listes numérotées, et

```xml
<list type="unordered">
  <item>...</item>
</list>
```

pour les listes à puces.

### Règles

- pas de paragraphes libres dans un `list` hors `item` ;
- pas de listes artificielles construites par tirets manuels ;
- les listes imbriquées sont hors périmètre V1, sauf si le validateur les autorise explicitement.

---

## 12. Tableaux simples

Les tableaux Word simples doivent produire une structure TEI tabulaire simple.

### Forme cible

```xml
<table>
  <row>
    <cell>Colonne 1</cell>
    <cell>Colonne 2</cell>
  </row>
  <row>
    <cell>Valeur A</cell>
    <cell>Valeur B</cell>
  </row>
</table>
```

### Règles

- utiliser exclusivement `<table>`, `<row>` et `<cell>` ;
- ne pas encoder de fusion de cellules en V1 ;
- ne pas créer d’attributs complexes de rendu ;
- ne pas tenter de reconstituer une sémantique plus fine que celle du tableau simple ;
- le contenu des cellules doit rester simple : texte, inline, éventuellement note si le pipeline le permet.

### Politique sur la première ligne

L’implémentation peut soit :

- traiter la première ligne comme une ligne ordinaire ;
- ou la signaler par un attribut léger si cela est stable et utile.

### Recommandation V1

Traiter toutes les lignes uniformément, sans logique d’en-tête spéciale.

---

## 13. Légendes

Le style `Légende` / `Caption` doit être converti en une structure simple.

### Recommandation V1

Utiliser :

```xml
<p type="caption">...</p>
```

### Règles

- ne pas introduire de structure `<figure>` complexe en V1 si l’objet illustré n’est pas lui-même pris en charge ;
- si une légende est isolée sans objet clairement associé, elle reste un paragraphe typé.

---

## 14. Bibliographie

La section bibliographique ne dépend pas d’un style Word propre, mais d’un **contexte structurel**.

### 14.1. Déclenchement

Une section est considérée comme bibliographique lorsqu’un titre de section a pour contenu normalisé :

- `Bibliographie`
- éventuellement d’autres formes approuvées par configuration

### 14.2. Forme cible

Dans cette section, les paragraphes bibliographiques consécutifs doivent être regroupés en :

```xml
<div type="section">
  <head>Bibliographie</head>
  <listBibl>
    <bibl>Forestier, ...</bibl>
    <bibl>Guibert, ...</bibl>
  </listBibl>
</div>
```

### 14.3. Règles

- `listBibl` n’apparaît que dans une section bibliographique ;
- chaque entrée devient un `<bibl>` ;
- ne pas encapsuler chaque entrée dans `<p>` à l’intérieur de `<bibl>` en V1, sauf contrainte technique forte ;
- les paragraphes vides sont ignorés ;
- l’ordre source doit être conservé.

---

## 15. Politique sur le texte libre au niveau racine

À l’intérieur de `<div type="notice">`, le texte libre ne doit apparaître que sous forme de :

- `<head type="main">`
- `<head type="sub">`
- `<p>`
- `<quote>`
- `<list>`
- `<table>`
- `<div>` de section
- `<p type="caption">`

Autrement dit, il ne doit pas y avoir de nœuds textuels nus directement sous le `div` de notice.

---

## 16. Politique sur la simplification

La cible TEI V1 doit rester simple.

### À éviter en V1

Ne pas produire automatiquement :

- `<front>` / `<back>` sophistiqués ;
- `<figure>` complexes ;
- `<cit>` au lieu de `<quote>` sans besoin documenté ;
- `<lb/>` pour mimer Word ;
- `<seg>` généralisés ;
- attributs `@rend` multiples et décoratifs ;
- attributs analytiques non stabilisés.

### Règle directrice

Privilégier la régularité des tests et la lisibilité du XML sur le raffinement sémantique prématuré.

---

## 17. Exemple complet de sortie cible

```xml
<TEI xmlns="http://www.tei-c.org/ns/1.0">
  <teiHeader>
    <fileDesc>
      <titleStmt>
        <title>Bérénice</title>
      </titleStmt>
      <publicationStmt>
        <p>Generated by Ekdosis–TEI Studio</p>
      </publicationStmt>
      <sourceDesc>
        <p>Imported from a styled Word notice.</p>
      </sourceDesc>
    </fileDesc>
  </teiHeader>
  <text>
    <body>
      <div type="notice" xml:id="berenice-notice">
        <head type="main">Bérénice</head>
        <head type="sub">Notice</head>

        <div type="section">
          <head>Genèse</head>
          <p>Premier paragraphe.</p>
          <p rend="noindent">Second paragraphe sans retrait.</p>
          <quote>
            <p>Texte cité.</p>
          </quote>
        </div>

        <div type="section">
          <head>Repères</head>
          <list type="unordered">
            <item>Premier point</item>
            <item>Deuxième point</item>
          </list>
          <table>
            <row>
              <cell>Édition</cell>
              <cell>Date</cell>
            </row>
            <row>
              <cell>Originale</cell>
              <cell>1670</cell>
            </row>
          </table>
        </div>

        <div type="section">
          <head>Bibliographie</head>
          <listBibl>
            <bibl>Forestier, Georges, ...</bibl>
            <bibl>Guibert, Albert-Jean, ...</bibl>
          </listBibl>
        </div>
      </div>
    </body>
  </text>
</TEI>
```

## 18. Invariants de test

Les tests automatiques doivent pouvoir vérifier au minimum les invariants suivants.

### 18.1. Invariants structuraux

Le XML généré contient :

- un `<TEI>` racine ;
- un `<teiHeader>` ;
- un `<text>` ;
- un `<body>` ;
- un seul `<div type="notice">` racine ;
- un seul `<head type="main">` au niveau du `div` de notice.

### 18.2. Invariants de mapping

Les tests doivent vérifier que :

- `Titre` -> `<head type="main">`
- `Sous-titre` -> `<head type="sub">`
- `Titre 1` -> `<div type="section">`
- `Titre 2` -> `<div type="subsection">`
- `Titre 3` -> `<div type="subsubsection">`
- `Titre 4` -> `<div type="level4">`
- style sans retrait -> `<p rend="noindent">`
- petites capitales -> `<hi rend="smallcaps">`
- note de bas de page -> `<note place="foot">`
- citation -> `<quote><p>...</p></quote>`
- tableau simple -> `<table>/<row>/<cell>`
- bibliographie -> `<listBibl>/<bibl>`

### 18.3. Invariants négatifs

Les cas V1 valides ne doivent pas produire spontanément :

- de texte nu sous `<div type="notice">` ;
- de `Heading 5` converti artificiellement ;
- de `<figure>` complexes ;
- de tableaux avec fusions simulées ;
- de styles Word non mappés silencieusement.

---

## 19. Résumé prescriptif

La chaîne d’import des notices doit produire un **document TEI complet**, structuré autour d’un `<div type="notice">`, avec :

- un titre principal en `<head type="main">` ;
- un sous-titre facultatif en `<head type="sub">` ;
- une hiérarchie de sections en `div` imbriqués ;
- des paragraphes simples en `<p>` ;
- le style sans retrait en `<p rend="noindent">` ;
- les petites capitales en `<hi rend="smallcaps">` ;
- les citations en `<quote>` ;
- les notes en `<note place="foot">` ;
- les listes en `<list>` ;
- les tableaux simples en `<table>` ;
- la bibliographie en `<listBibl>`.

Cette cible TEI constitue le contrat de sortie de la V1. Toute implémentation doit s’y conformer de manière stable et testable.
